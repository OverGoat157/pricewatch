## 1. Введение

Цены на товары в маркетплейсах меняются по несколько раз в неделю: действуют
акции, распродажи, динамическое ценообразование. Покупателю, который ждёт
выгодную цену на конкретный товар, неудобно вручную заходить на страницу и
проверять её каждый день. Возникает потребность в инструменте, который сам
следит за ценой и сообщает, когда она опустилась до нужного уровня.

Цель работы — разработать веб-приложение «PriceWatch» для мониторинга цен на
товары Wildberries: автоматический сбор цены (парсинг), хранение истории
изменений с визуализацией и отправку уведомлений в Telegram при достижении
заданной пользователем целевой цены.

## 2. Постановка задачи

**Цель:** реализовать работоспособный веб-сервис мониторинга цен.

**Задачи:**

1. Спроектировать и реализовать REST API на Go.
2. Спроектировать реляционную модель данных в PostgreSQL.
3. Реализовать парсер цен с маркетплейса Wildberries.
4. Реализовать фоновый планировщик периодической проверки цен.
5. Реализовать детектирование снижения цены и отправку уведомлений в Telegram.
6. Реализовать многопользовательский режим с регистрацией и авторизацией (JWT).
7. Разработать одностраничное веб-приложение (SPA) на Vue 3 с графиком истории цен.
8. Обеспечить развёртывание всей системы через Docker Compose.

**Функциональные требования:**

- регистрация и вход пользователя;
- добавление товара по ссылке/артикулу и указание целевой цены;
- просмотр списка отслеживаемых товаров с текущей ценой и статусом;
- просмотр истории цен товара в виде графика и таблицы;
- автоматическая и ручная («Проверить сейчас») проверка цены;
- уведомление пользователя при снижении цены до целевой.

**Нефункциональные требования:**

- изоляция данных между пользователями;
- хранение паролей в виде хешей;
- устойчивость к недоступности внешнего источника (ошибка парсинга не должна ронять сервис);
- воспроизводимое развёртывание одной командой.

## 3. Обзор предметной области

Существуют сервисы отслеживания цен (например, расширения и сайты-агрегаторы
истории цен маркетплейсов). Их общие черты: подписка на товар, график истории,
оповещение при снижении. На основе анализа выделены ключевые сущности
предметной области: **пользователь**, **товар**, **подписка с целевой ценой**,
**история цен**, **уведомление**. Эти сущности легли в основу модели данных.

Источником цен выбран Wildberries: у него есть публичный JSON-эндпоинт карточки
товара (`card.wb.ru`), что надёжнее, чем разбор HTML-страницы (HTML-вёрстка
часто меняется и защищена от ботов).

## 4. Выбор инструментов разработки

| Компонент   | Технология | Обоснование |
|-------------|-----------|-------------|
| Язык бэкенда | **Go** | Производительность, простая многопоточность (планировщик и бот работают в отдельных горутинах), статическая типизация, единый бинарник. |

Стек соответствует профилю программы «Фронтенд и бэкенд разработка»: Go +
PostgreSQL на бэкенде, Vue на фронтенде.

## 5. Проектирование системы

### 5.1. Архитектура

Система состоит из трёх контейнеров:

```
Браузер ──▶ nginx (frontend, отдаёт SPA и проксирует /api)
                     │
                     ▼
              Go backend (REST API :8080)
                     │  ├─ Scheduler  — горутина: периодическая проверка цен
                     │  ├─ Parser     — обращение к Wildberries JSON API
                     │  └─ TelegramBot — горутина: long polling + отправка
                     ▼
                PostgreSQL
```

Бэкенд разбит на слои: `api` (HTTP-обработчики) → `store` (доступ к данным) →
PostgreSQL. Бизнес-логика проверки цен вынесена в пакет `scheduler`, парсинг —
в пакет `parser` (за интерфейсом, что позволяет добавлять новые источники без
изменения остального кода).

### 5.2. Модель данных

Пять таблиц (денежные значения хранятся в копейках, тип BIGINT, для точности):

- **users** — пользователи (email, хеш пароля, telegram_chat_id, link_token);
- **products** — общий каталог товаров (источник, артикул, название, последняя цена); уникальность по (source, external_id), чтобы не парсить один товар многократно;
- **subscriptions** — подписка пользователя на товар с целевой ценой; связь «многие-ко-многим» между users и products;
- **price_history** — точки истории цены товара (цена, наличие, время);
- **notifications** — журнал отправленных уведомлений.

**Связи (ER):**

```
users (1) ──< (N) subscriptions (N) >── (1) products
products (1) ──< (N) price_history
users (1) ──< (N) notifications >── (1) products
```

Все внешние ключи объявлены с `ON DELETE CASCADE`. Созданы индексы по
`price_history(product_id, checked_at)` и `notifications(user_id, sent_at)`.

### 5.3. Проектирование API

REST-эндпоинты сгруппированы: публичные (`/api/auth/*`) и защищённые
(требуют заголовок `Authorization: Bearer <JWT>`). Полная таблица — в README и
разделе 8.

## 6. Реализация

### 6.1. Парсинг цены (пакет `parser`)

Определён интерфейс `Parser` с методами `ExternalID` (извлечь артикул из
ссылки) и `Fetch` (получить цену). Реализация для Wildberries обращается к
`https://card.wb.ru/cards/v2/detail` и разбирает JSON. Поскольку формат ответа
WB со временем менялся, цена определяется устойчиво — по нескольким полям в
порядке приоритета (итоговая цена со скидкой → цена карточки → устаревшие поля).
Артикул извлекается регулярным выражением как из полной ссылки
(`catalog/<артикул>/detail.aspx`), так и из «голого» числа.

### 6.2. Фоновая проверка и детект снижения (пакет `scheduler`)

Планировщик в отдельной горутине по тикеру (интервал настраивается) обходит все
товары: получает цену, записывает точку в `price_history`, обновляет последнюю
цену. Уведомление отправляется только в момент **пересечения** целевой цены
сверху вниз: текущая цена ≤ цели, а предыдущая была выше. Это исключает
повторный спам, пока цена держится низкой.

### 6.3. Авторизация (пакет `auth`)

Пароли хешируются bcrypt. При входе выдаётся JWT (HS256) со сроком жизни и
идентификатором пользователя. Middleware проверяет токен и кладёт `userID` в
контекст запроса; все запросы к данным фильтруются по этому идентификатору —
так обеспечивается изоляция данных пользователей.

### 6.4. Уведомления в Telegram (пакеты `notify`, `telegrambot`)

Привязка чата: пользователь генерирует одноразовый код, открывает бота по
ссылке `t.me/<bot>?start=<код>`. Бэкенд через long polling (`getUpdates`)
получает команду `/start <код>`, находит пользователя по коду и сохраняет
`chat_id`. Далее уведомления отправляются методом `sendMessage`.

### 6.5. Фронтенд (Vue 3)

SPA из страниц: вход/регистрация, дашборд со списком товаров, страница товара
с графиком (Chart.js) и таблицей истории, настройки (привязка Telegram).
Состояние авторизации хранится в Pinia, токен — в localStorage и
подставляется в каждый запрос перехватчиком Axios. Маршрутизация защищена
навигационным guard'ом.

## 7. Тестирование

### 7.1. Модульное тестирование

Логика разбора ответа Wildberries покрыта юнит-тестами (пакет `parser`):
проверка извлечения артикула из разных форматов ссылок и корректность разбора
цены/названия/наличия на сохранённом JSON-ответе (фикстуре). Тесты не зависят
от сети, запускаются командой `go test ./...` — все проходят успешно.

### 7.2. Функциональное тестирование (ручное / API)

После развёртывания через Docker Compose проверены сценарии:

## 8. Руководство пользователя

1. Скопировать `.env.example` в `.env`, при необходимости указать токен Telegram-бота.
2. Выполнить `docker compose up --build`.
3. Открыть `http://localhost:8080`, зарегистрироваться.
4. На дашборде вставить ссылку на товар Wildberries и целевую цену → «Отслеживать».
5. Открыть товар, чтобы увидеть график истории; кнопка «Проверить сейчас» обновляет цену.
6. В «Настройках» привязать Telegram, чтобы получать уведомления о снижении цены.

## 9. Результаты

Разработано работоспособное веб-приложение, выполняющее все поставленные
задачи: многопользовательский мониторинг цен с парсингом Wildberries,
хранением и визуализацией истории и уведомлениями в Telegram. Система
разворачивается одной командой через Docker Compose; работоспособность
подтверждена модульными и функциональными тестами.

В ходе работы спроектирована и реализована полноценная клиент-серверная

реализация авторизации (JWT, bcrypt), разработка SPA на Vue 3 и контейнеризация
